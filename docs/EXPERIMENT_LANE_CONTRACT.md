# Experiment Lane Contract

## Purpose

This document defines the contract for the bounded experiment lane.

The lane exists to add a fast, reproducible experiment capability without introducing:

- a new global orchestrator
- a new source of truth
- uncontrolled autonomy
- contract drift between June and Operator

The lane is a subordinate execution capability.
June remains the only global orchestrator.
Operator remains the only epistemic source of truth.

## Non-Negotiable Rules

The experiment lane must obey these four rules:

1. no new orchestrator
2. no new truth layer
3. worker writes bounded artifacts only
4. Operator ingests, June decides

If any implementation violates one of these rules, it is out of contract.

## Authority Model

### June

June is the only global orchestrator.

June owns:

- mission identity
- mission-level dispatch decisions
- retry, replan, escalation, and stop decisions
- mission state for experiment dispatch

June does not own experiment truth.

### Operator

Operator is the only epistemic source of truth.

Operator owns:

- project identity and lifecycle
- experiment instantiation within a project context
- experiment result ingestion
- project-level state transitions
- state-relevant control-plane events derived from experiment outcomes

If June and Operator disagree on experiment truth, Operator wins.

### Experiment Worker

The experiment worker is a bounded execution subsystem.

The worker owns:

- experiment-local run execution
- experiment-local artifacts
- experiment-local metrics extraction
- experiment-local keep/discard decisions inside the bounded loop

The worker does not own mission truth, project truth, or global next action.

## Identity Contract

The experiment lane uses three canonical identifiers:

- `mission_id`: canonical June mission identifier
- `project_id`: canonical Operator project identifier
- `experiment_id`: canonical bounded experiment-lane identifier

Rules:

- an experiment must be bound to exactly one `mission_id`
- an experiment must be bound to exactly one `project_id`
- an experiment must not exist without both bindings
- `experiment_id` is generated by Operator
- June must reference the bound `experiment_id`; June must not generate a competing experiment identity
- retries do not create a new `experiment_id` unless June and Operator explicitly authorize a new experiment instance

## Dispatch Model

The canonical dispatch path is:

User or internal approved task -> June -> Operator experiment instantiation -> bounded worker execution -> Operator ingest -> June next decision

Rules:

- all externally meaningful work still enters through June
- Operator may instantiate an experiment only within a project already owned by Operator
- the worker may execute only after Operator has produced a canonical experiment brief
- the worker must never self-promote into a global continuation loop

## Canonical Artifacts

The bounded worker may produce only experiment-local artifacts.

Required artifacts:

- `experiment_brief.json`
- `experiment_trace.jsonl`
- `experiment_result.json`

Optional artifacts:

- `runs/<run_id>/run.log`
- `runs/<run_id>/metrics.json`
- `runs/<run_id>/diff.patch`
- other run-local files explicitly allowed by the experiment brief

The worker must not directly mutate canonical project truth files such as `project.json`.

## Experiment Brief Contract

`experiment_brief.json` is the only canonical worker input.

Required fields:

- `mission_id`
- `project_id`
- `experiment_id`
- `owner`
- `dispatch_owner`
- `hypothesis`
- `objective`
- `editable_paths`
- `read_only_paths`
- `run_command`
- `parse_metric`
- `metric_name`
- `metric_direction`
- `time_budget_seconds`
- `max_runs`
- `acceptance_rule`
- `revert_rule`
- `termination_conditions`
- `baseline`

Required defaults:

- `owner` = `operator`
- `dispatch_owner` = `june`
- `metric_direction` must be exactly `min` or `max`

Rules:

- the worker may edit only paths in `editable_paths`
- the worker must treat all other paths as immutable
- the worker must fail closed if the brief is missing required fields
- the worker must fail closed if the metric cannot be parsed under `parse_metric`

## Event Contract

The following experiment-lane event types are allowed:

- `experiment_dispatched`
- `experiment_run_completed`
- `experiment_improved`
- `experiment_inconclusive`
- `experiment_failed`

### Event emitter rules

June may emit:

- mission-level dispatch context related to experiment dispatch

Operator may emit:

- state-relevant experiment outcome events
- control-plane events derived from ingested experiment results

The worker may emit:

- experiment-local trace events only

The worker must not emit:

- mission decisions
- global control-plane decisions
- project-terminal truth events

## Event Payload Schema

All experiment-related control-plane events must include these required fields:

- `mission_id`
- `project_id`
- `experiment_id`
- `control_plane_owner`
- `handoff_source`
- `dispatch_cause`
- `timestamp`

All experiment result events must additionally include:

- `status`
- `metric_name`
- `metric_direction`
- `baseline_value`
- `best_value`
- `runs_attempted`
- `terminal_reason`

Optional fields:

- `best_run_id`
- `artifact_path`
- `summary`
- `failure_class`

Rules:

- `status` must use only contract-approved result semantics
- `best_value` may equal `baseline_value` only for `inconclusive`
- `artifact_path` must point to experiment-local artifacts only
- `terminal_reason` must be populated for every terminal experiment result event

## Result Semantics

The experiment lane recognizes exactly four terminal result states:

- `improved`
- `inconclusive`
- `failed`
- `invalid`

Definitions:

- `improved`: the metric improved against the baseline according to `metric_direction` and `acceptance_rule`
- `inconclusive`: the experiment ran validly but did not improve against the baseline
- `failed`: the experiment could not complete operationally, for example due to crash, timeout, or environment failure
- `invalid`: the result is technically or epistemically unusable, for example due to missing metric, malformed output, or contract violation

Hard rule:

- no worker result is global truth on its own
- Operator must ingest and interpret the result before it affects project truth
- June must decide the next mission action after Operator has produced a machine-readable outcome

## Write Boundaries

### Worker may write

- experiment-local artifacts
- trace logs
- run logs
- run metrics
- diff/patch artifacts
- worker-local temporary state explicitly scoped to the experiment workspace

### Worker may not write

- `project.json`
- mission state files
- project terminal status
- global control-plane truth
- user-facing mission conclusions

### Operator may write

- project state updates derived from experiment ingestion
- experiment references in project state
- state-relevant control-plane events
- epistemic outcome summaries

### June may write

- mission dispatch state
- retry / replan / escalation decisions
- mission-level conclusions based on Operator-ingested results

## Idempotency and Retry

The canonical idempotency key for the experiment lane is `experiment_id`.

Rules:

- a dispatch for an already-active `experiment_id` must not create a second active worker
- retries within the same experiment instance must reuse the same `experiment_id`
- a new `experiment_id` requires explicit June and Operator approval for a new experiment instance
- crash recovery must inspect existing experiment artifacts before deciding whether to resume or fail closed
- duplicate dispatch detection must happen before worker start

## Validation Rules

A conforming implementation must satisfy all of the following:

- no second orchestrator emerges from the experiment lane
- no project truth is created outside Operator
- the worker is bounded by explicit budget and scope
- all experiment state is attributable to `mission_id`, `project_id`, and `experiment_id`
- event types and payloads stay within this contract
- retries and duplicate dispatches are idempotent under `experiment_id`

## Out of Scope for V1

The following are explicitly out of scope for the first experiment-lane implementation:

- ARGUS and ATLAS ownership inside the lane
- portfolio-level experiment scheduling
- open-ended autonomous self-expansion
- replacing the existing research lifecycle with the experiment lane
- any parallel mission-truth or project-truth model

## Acceptance Standard

This contract is complete when both sides can implement against it without making new decisions about:

- authority
- identity
- event names
- required payload fields
- write boundaries
- idempotency behavior
- terminal result semantics
